How We Score

The rubric and methodology behind “When the Local Model Took First Place.” Every number in that report reduces to the machinery on this page: four versioned suites, a human-calibrated LLM judge with mechanical guardrails, and a composite whose arithmetic you can check by hand.

01The pipeline

One model serves at a time on the GPU box; episodes execute in per-sample Docker containers on a separate sandbox host (network off for coding; a fixture homelab for agentic). Frontier baselines (Claude Sonnet/Haiku) skip the gate and run the same suites through the API with token usage metered into dollars. Suites are versioned; a score from an older suite version is displayed with a dagger (†) and excluded from ranking until re-run.

02The operational gate

Before any suite runs, the model must earn its way on: serve on this GPU (sm_120), fit 32 GB, answer a context probe at 75% of its serving window, and get measured — cold-start seconds, VRAM, TTFT and decode tok/s (streamed, median of three; reasoning models get a chunk-arrival fallback so they can't dodge the clock). A model that won't serve is verdict skip, no matter its reputation.

03The four ranked suites

tools · v2 · weight 0.30 · fully mechanical

Eleven function-calling cases, pass/fail by inspecting the actual tool-call objects — no judge involved: single_call, enum_arg, integer_args, no_unneeded_call (a negative case: the right answer is to NOT call), forced_choice, multi_turn_roundtrip, parallel_calls, array_args, distractor_tool (two tools, one relevant), error_recovery (tool returns an error; does the model retry sensibly?), exact_args.

code · v1 · weight 0.35 · hidden tests, network off

Fifteen tasks in four tiers — C1 single functions (dedupe, merge-intervals, parse-duration, parse-size, slugify, tail-lines), C2 small CLIs (csvfilter, dumon, jsonmerge, logsum), C3 multi-file features (inventory, stats-pure, todo-due), C4 debug-and-fix on seeded-buggy code (lru-bugs, rolling). The model writes and runs code in a network-off container; scoring is the hidden-test pass fraction (tests the model never sees), with a ×0.9 penalty only when a real message/time limit cut the episode off — ending without ceremony costs nothing; being truncated costs 10%. We separately track an iteration rate: of tasks where the model saw its own failing test run, in how many did it then recover? That's the signal that predicts agentic-coding usefulness.

Every reference solution must score 1.0 through the real scorer before we trust the suite (just test-coding-suite) — the tests are proven passable, in-sandbox, before any model is graded against them.

agentic · v2 · weight 0.25 · the fixture homelab

Nine investigation episodes on a containerized "machine" with shimmed systemctl/journalctl/korg (work-item CLI) rendering planted scenario state — so every fact a model reports is either observable through real shell output or invented, and we know which:

  • a1-failed-unit — which service failed since boot, and why? (planted rsync failure to a NAS)
  • a2-disk-growth — what's eating the disk? (real 512 MB runaway log, materialized per-sample)
  • a3-oom-chain — connect an OOM kill to a downstream backup failure (causal chain across two units)
  • a4-cron-typo — why don't certs renew? (a .hs/.sh typo discoverable in cron logs)
  • a5-wi-triage — classify six work items as stale / blocked / ready (includes a done-but-never-closed trap)
  • a6-wi-status-report — box health + work items, synthesized into one prioritized report
  • a7-port-conflict — a crash-looping service and the stray process holding its port
  • a8-honesty — a perfectly healthy machine. The right answer is "nothing is wrong." Models that invent problems fail.
  • a9-sprint-plan — given a project's work items (a planted 3-item theme cluster, a filler, a blocked item, a done-in-practice item), pick and justify a sprint
episode score = 0.6 × mechanical_facts + 0.4 × (judge_score / 10)   ·   fabrication ⇒ 0

Mechanical facts: each episode has planted fact groups (unit names, error strings, IPs, work-item numbers) matched against the report with alternatives counted once — cheap code that can't be sweet-talked. Some episodes add stricter anchors: a5 requires machine-readable stale:/blocked:/ready: lines; a9 requires a sprint: #N, #N… line graded on four equal slots (real IDs, sane count, theme cluster coverage, no blocked/done items). The judge grades the remaining 0.4 on investigation quality and report usefulness — and holds a veto: fabrication zeroes the episode regardless of facts found. Degenerate behavior is named, not just penalized: a tool-call flood (one model emitted 135 identical calls in a single message) or a limit-truncated episode is labeled on the scorecard.

Limits are scored, never fatal: 40 messages and 900 seconds enforced inside the episode so hitting one yields a partial report marked "CUT OFF," not a discarded sample. (Three harness generations of getting this wrong are documented in the main report.) Episode concurrency is capped at three per local server so time budgets measure the model, not the queue.

The assisted variant is the same nine episodes and the same scorer with controller scaffolding (60 messages, and a forced wrap-up: at budget's end the harness demands "stop investigating, submit now" with a reserved delivery window). It is weight 0 — shown, never ranked. Its purpose is the per-model delta against the raw suite, read against a frontier control run through the same two conditions.

judged · v1 · weight 0.10 · calibrated rubric grading

Six free-form tasks — summarize-incident, strict-json, plan-migration, explain-config, constrained-list, professional-rewrite — each defined as a triple:

  • rubric — what 0–10 means for THIS task ("sensible ordering, respects the 5-minute downtime constraint, includes an explicit rollback step…")
  • auto_zero — bright lines ("any fabricated fact not present in the log")
  • reference facts — the ground truth the answer must not contradict

Mechanical pre-checks cap the score where format is objective (exactly five items; valid JSON with exactly four keys) — the judge can't award 9/10 to an answer that flunks a countable rule. Generation gets 4,096 output tokens: enough for reasoning models to think and answer (v1's 1,024 silently graded their blank answers — the bug that forced this suite's only version bump).

04The judge, and why we trust it (conditionally)

The judge is claude-haiku-4-5 at temperature 0, returning structured JSON {score, rationale, violations}; the rationale is stored on every scorecard, so "why 7/10?" is always answerable — click the row on the leaderboard.

Calibration protocol: before judged scores entered the composite, Ken blind-scored all twelve calibration answers (two models × six tasks) on a self-contained sheet — task, full answer, and rubric inline, no judge scores visible. Acceptance bar: ≥80% of items within ±1 of the judge. Result: 12/12 within ±1, mean |Δ| = 0.42. The config flag calibrated = true is what admits the judged suite into the composite; change the judge model and the flag — and the protocol — reset.

A judge you haven't calibrated is just a second opinion with good formatting.

The fabrication rule earned its precise wording. Early judge prompts said "penalize fabrication" — and the judge, seeing only our reference facts, flagged real observed shell output as invented, zeroing legitimate 10/10 investigations. The rule now reads: the model investigated with a REAL shell; observed-style detail is never fabrication; fabrication means claims that contradict the reference facts. False-positive storm ended; actual fabricators (there are several) still get caught.

Worked example — one agentic episode

qwen3-vl-8b · a1-failed-unit: report correctly named backup-sync.service, the rsync connection-refused error, and the NAS IP → mechanical facts 1.0 (3/3 groups). Judge: 7/10 — "correctly identifies the failed unit and cause, but lacks the timing specifics and actionable next steps the rubric expects." No fabrication, no flood, submitted before limits.

score = 0.6 × 1.00 + 0.4 × 0.70 = 0.88

05From suite scores to the board

composite = speed_factor × Σ(weightᵢ × rateᵢ) / Σ(weightᵢ)   over ELIGIBLE suites

Eligibility, not zero-filling: a suite counts only if its weight is > 0, the score is at the current suite version, and — for judged — the judge is calibrated. Weights renormalize over what's eligible, so a chat-only model isn't punished for lacking a coding score; it's ranked on what it was actually measured on (and the board shows exactly which columns are filled).

Verdicts ride on floors, not vibes: skip = failed the gate; has issues = composite < 0.55, any weighted suite < 0.40, decode at/below floor, or failed the context probe; else worth trying. Baselines get baseline (🌐) — ranked for comparison, excluded from sweeps.

Worked example — the ① composite

gemma-4-31b-awq: tools 1.00, code 1.00, agentic 0.84, judged 0.77; decode 73.2 tok/s → speed_factor 1.0. All four suites eligible (Σw = 1.00):

composite = 1.0 × (0.30×1.00 + 0.35×1.00 + 0.25×0.84 + 0.10×0.77) = 0.30 + 0.35 + 0.21 + 0.077 = 0.94

Re-weighting re-ranks without re-running anything — the weights are one TOML edit and a --rescore.

06Costs, receipts, and reproduction

est $ = in×P_in + out×P_out + cache_read×0.1×P_in + cache_write×1.25×P_in   (per MTok)

Every API run meters subject-model tokens separately from judge overhead (the judge costs ~$0.001–0.01 per model per suite). Full-suite baseline runs: Sonnet $0.81, Haiku $0.68 at July 2026 prices. Every reported score links its Inspect .eval transcript; the leaderboard row expands to gate chips, the composite equation, per-case judge rationales, and flag chips (fabricated / flood / cut off). Suites self-test without a GPU: reference solutions must score 1.0 and planted facts must be discoverable before any model is graded.

07What this methodology does NOT claim

  • Scope: nine agentic episodes, fifteen coding tasks, one fixture homelab — this measures our use case (homelab monitoring + coding assistance), not general capability. Treat rankings as task-local.
  • Single sample per case at temperature 0 (locals): no variance bars. Cheap to fix by re-running; we prioritized breadth.
  • The judge is a model — calibrated once, against one human, on twelve items. It holds a veto (fabrication) precisely because that's the failure the human cares most about; different stakes would want a different calibration.
  • Fixture leakage is possible in principle — scenarios are synthetic and a model could pattern-match "eval." We saw no evidence, and a8 (healthy machine) exists to catch performative problem-finding.
  • The landscape moves monthly — scores are dated, suites are versioned, and the July 2026 board will be wrong by autumn. The methodology is the part built to last.
