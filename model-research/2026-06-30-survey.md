# Model survey — best open models for a 32 GB RTX 5090 on vLLM (2026-06-30)

_Broad sweep across coding, agentic tool-use, and vision. Produced by the deep-research harness
(fan-out search → fetch → 3-vote adversarial verify); the final synth stage crashed, so this report
was synthesized by hand from the **75 verified votes / 121 extracted claims** in the run journal.
Confidence and caveats below reflect the adversarial pass — including several claims it **refuted**._

> Standing constraints live in [`README.md`](README.md). TL;DR: single RTX 5090, 32 GB, sm_120;
> vLLM 0.24.0 (safetensors; FP8/AWQ/GPTQ; not GGUF); tool calling required; leave KV-cache headroom.

## Headlines

- **Qwen3.6 is real** (Ken's question). Two open-weight, Apache-2.0, **ungated** members — verified
  against the live HuggingFace API, not just a blog:
  - **`Qwen/Qwen3.6-27B`** — dense 27B, **multimodal** (image+video→text), thinking-mode default,
    released ~2026-04-22. Billed "flagship-level coding in a 27B dense model."
  - **`Qwen/Qwen3.6-35B-A3B`** — MoE 35B total / **3B active**, 256 experts, multimodal, ~2026-04-16.
  - ⚠️ Both use a **new architecture** (`model_type` = `qwen3_5` / `qwen3_5_moe`, hybrid Gated-DeltaNet
    + Gated-Attention + vision encoder). The blog's "Qwen 3.6 **Plus**/Max" are **proprietary hosted
    SKUs**, *not* these open weights. Official serve example uses `--tensor-parallel-size 8` (multi-GPU)
    at 262K context → for one 5090 you need a **4-bit AWQ** build + reduced context, **and** you must
    confirm vLLM 0.24.0 / sm_120 actually supports `qwen3_5*` before trusting it. **Promising but
    bleeding-edge — verify before relying on it.**

- **Best coding upgrade: Qwen3-Coder-30B-A3B** (MoE 30.5B/3.3B active, Apache-2.0, agentic-trained,
  Qwen Code / CLINE / Claude Code). ⚠️ **The official FP8 build (~31.2 GB weights) does NOT fit a
  32 GB 5090** — it OOMs with no KV-cache room (verify-refuted, high confidence; the FP8 build is for
  ≥48 GB cards). **Use a 4-bit AWQ build (~18–22 GB).** Tool parser: **`qwen3_xml`**, *not* the
  eponymous `qwen3_coder` (which has an infinite-`!!!!` token bug — vLLM #22975).

- **Best agentic-coding peer: Devstral-Small-2-24B** (`mistralai/Devstral-Small-2-24B-Instruct-2512`,
  Apache-2.0) — **68.0% SWE-Bench Verified, matching the 355B GLM-4.6**, purpose-built for SWE agents.
  Tool parser **`mistral`**. Fits via FP8/AWQ (bf16 is ~48 GB; the shipped FP8 reportedly dequants to
  bf16, so a 4-bit AWQ build is the reliable single-GPU path).

- **Two vLLM-flag corrections for our environment (0.24.0):**
  - `--enable-reasoning` is **removed** (deprecated 0.9.0, gone 0.10.0). Enable thinking with
    `--reasoning-parser <name>` **alone**.
  - For **Qwen3** reasoning the modern parser is **`qwen3`**, not `deepseek_r1` (which still works but
    is the older fallback; current vLLM maps the Qwen3 series → `qwen3`).

## sm_120 / Blackwell quantization guidance (cross-cutting)

| Format | On sm_120 (RTX 5090) | Use it for |
|---|---|---|
| **AWQ (awq_marlin)** | **Proven workhorse.** ~140 tok/s with CUDA graphs (vs ~17 enforce-eager). | 14B–35B tier; the way to fit big models in 32 GB. |
| **FP8 W8A8** | **Native on Blackwell** (cc > 8.9), runs via FlashInfer kernel — fast on **native Linux** (the "FP8 slow" reports were WSL-only, where dxgkrnl hides the FP8 cores). | 7–14B where you want quality + speed. |
| **GPTQ (gptq_marlin)** | Works; comparable to AWQ. | When the checkpoint ships GPTQ. |
| **INT8 / bitsandbytes** | ❌ **Corrupted output on sm_120.** Avoid. | — |
| **NVFP4** | Bleeding-edge: native kernels compiled but vLLM's backend-selection guard excluded `(12,0)` as of early 2026 → silently falls back to slower Marlin. | Revisit later. |

Other sm_120 notes: **CUDA graphs work** (don't use `enforce-eager`); **flash-attn can hit
undefined-symbol** → `--attention-backend flashinfer` is the safe fallback (though our Sprint-1 run
got FlashAttention-2 auto-selected fine on 0.24.0 — version/setup dependent); **`--kv-cache-dtype
fp8`** stretches context to fit big models in 32 GB.

> Streaming caveat (relevant to our "non-streaming tool turns first" choice): a hermes **streaming**
> tool-call bug (raw `<tool_call>` XML in `content`) was reported on vLLM 0.13.0 (#31871) and marked
> fixed ~2026-05. The verifier judged the "streaming always breaks" framing **overstated/version-
> pinned** — so on 0.24.0 streaming tools may be fine, but non-streaming remains the safe default.

## Ranked shortlist

### Coding
1. **Qwen3-Coder-30B-A3B (AWQ 4-bit)** — agentic coding star; `qwen3_xml` + `qwen3` reasoning; ~18–22 GB;
   256K ctx (drop to 32K if tight). _Action: locate an AWQ build — official is FP8-only and OOMs._
2. **Devstral-Small-2-24B-2512** — `mistralai/Devstral-Small-2-24B-Instruct-2512`; `mistral` parser;
   SWE-Bench Verified 68%; AWQ ~14 GB / FP8 ~24 GB. **Supersedes `qwen2.5-coder-7b` for serious work.**
3. **Qwen3.6-27B (AWQ)** — `Qwen/Qwen3.6-27B` (or `QuantTrio/Qwen3.6-27B-AWQ`); multimodal + "flagship
   27B coding"; `qwen3_xml`/`qwen3_coder` + `qwen3`. ⚠️ new arch — verify sm_120/0.24.0 support.

### Agentic / general tool-use
4. **Qwen3-8B (FP8)** — `Qwen/Qwen3-8B-FP8`; `hermes` + `qwen3` reasoning; ~9 GB; native 32K ctx.
   Strong general agent; **a newer peer to `llama-3.1-8b`.**
5. **Qwen3-14B (AWQ)** — bigger general agent, ~140 tok/s on the 5090; `hermes` + `qwen3`.
6. **Qwen3.6-35B-A3B (AWQ)** — `QuantTrio/Qwen3.6-35B-A3B-AWQ`; MoE, ~18–20 GB, multimodal, 256K ctx.
   ⚠️ same new-arch caveat. (Qwen3.5-35B-A3B GPTQ-Int4 also runs — ~194 tok/s — but **no documented
   tool-calling**, so weaker for agentic use.)

### Vision / multimodal
7. **Qwen3-VL-8B-Instruct** — `Qwen/Qwen3-VL-8B-Instruct`; ~17 GB FP16 (fits comfortably; AWQ/FP8
   available); **visual agent / GUI control + tool use**, 256K ctx. Top general vision pick.
8. **InternVL3-8B** — `OpenGVLab/InternVL3-8B`; best for **documents/OCR/tables/charts**; ~17 GB FP16;
   needs `--trust-remote-code`.
   - _(Qwen3.6-27B is itself multimodal if its arch serves. **Llama-4-Scout excluded** — 218 GB FP16 /
     ~55 GB INT4, does not fit.)_

## Method & confidence

108 agents, ~2.05 M tokens, 557 tool calls; 75 verified votes (66 confirmed, 9 refuted). The refuted
votes are load-bearing here — they corrected the FP8-fits-32 GB error, the removed `--enable-reasoning`
flag, and the Qwen3 reasoning-parser. Every model id above was seen in a primary source (HF API /
model card / vLLM docs); the **bleeding-edge `qwen3_5*` arch support on vLLM 0.24.0 + sm_120 is the
biggest unverified risk** and must be confirmed empirically on `kai` before these are marked `tested`.
